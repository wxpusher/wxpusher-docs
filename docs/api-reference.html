WxPusher 标准消息推送 API

由 WxPusher 维护 · 最后更新：2026-07-11 · 下载 OpenAPI 规范

标准推送适合把 WxPusher 集成进业务系统：应用用 appToken 鉴权，通过 UID 给用户单发，或通过 Topic 向订阅者群发。

最快接入路径：登录管理后台创建应用 → 获取 appToken → 让用户关注并取得 UID → 调用发送接口。appToken 属于密钥，不要提交到公开仓库。

发送消息

POST https://wxpusher.zjiecode.com/api/send/message，请求类型为 application/json。

curl -X POST https://wxpusher.zjiecode.com/api/send/message \
  -H 'Content-Type: application/json' \
  -d '{
    "appToken": "AT_xxx",
    "content": "服务器 CPU 使用率超过 90%",
    "summary": "服务器告警",
    "contentType": 1,
    "uids": ["UID_xxx"]
  }'

核心字段

字段说明
appToken应用鉴权密钥，必填。
content消息正文，必填。
contentType1 文本、2 HTML、3 Markdown。
uids接收用户 UID 数组，用于单发或批量发送。
topicIds主题 ID 数组，用于向主题订阅者群发；可与 uids 同时使用。
summary通知摘要，可选，最长 100 字符。
url消息原文链接，可选。

返回结果与后续操作

业务响应码 1000 表示成功。返回的 sendRecordId 可用于查询单个接收目标的发送状态；messageContentId 可用于删除消息落地页内容。消息为异步分发，接口成功表示发送任务已创建。

标准推送适用场景

服务器与运维告警

监控系统通过 UID 通知值班人员，或通过 Topic 通知团队。

业务事件通知

发送订单状态、任务结果、审核进度和信息更新提醒。

限制与权威文档

当前文档列出的主要限制包括：正文最长 40000 字符、单次最多 2000 个 UID 或 5 个 Topic，默认接口限流约 2 QPS。线上策略可能调整，请查看完整限制说明和完整请求与响应字段。用户关注、付费状态和上行消息等业务事件由 WxPusher 向应用配置的回调地址发起 HTTP 请求，具体字段以回调说明为准。
